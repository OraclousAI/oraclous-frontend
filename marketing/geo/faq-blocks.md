## Platform capability pages

### Harness & OHM (`/platform/harness-model`) — 4 Q
1. **What is a Harness?** — Oraclous's unit of work — a goal-driven assembly of human and Agent Actors under one policy envelope, holding a goal statement, Actor roster, orchestration spec, triggers, task-board reference, and provenance link. Every executable thing is a Harness, governed at the Harness level.
2. **What is OHM?** — The Oraclous Harness Manifest, the serialised, portable form of a Harness — YAML with embedded Markdown in `.ohm.yaml`. A structured zone of machine-validated fields and a prose zone of model-interpreted instructions; code always wins over prose. The canonical hub every portability operation routes through.
3. **How is a Harness different from a workflow?** — A workflow is a coded pipeline of steps; a Harness is a goal described in prose that the platform runs, governs, budgets, and audits as one object. No separate workflow concept — composition is a Harness referencing other Capabilities; humans are first-class Actors, not an exception path.
4. **Can a Harness contain other Harnesses?** — Yes. A Harness can reference other Capabilities including other Harnesses, so Harnesses compose. No separate "sub-workflow" object; governance, audit, and budget apply at each Harness level (ADR-003).

### Actors (`/platform/actors`) — 4 Q
1. **Can humans and AI agents share a task board?** — Yes — the core design. Humans and Agents are symmetric Actors with a common interface, so one Harness's task board holds work for both. The runtime dispatches Agents into a tool-use loop and humans into task-board assignments, no privileged code path, one governance model.
2. **What is the difference between an Actor and an Agent?** — An Actor is any entity that can be assigned work — human or AI. An Agent is the non-human kind, with its own identity, role, capability allocation, scope, and Consciousness record. An Agent is not the LLM; the LLM is a resource the Agent uses. Every Agent is an Actor; not every Actor is an Agent.
3. **How does work hand off between a human and an agent?** — Both ways, on the same task board. An Agent can finish a step and assign the next to a person; a person can pass control back. When a step needs a human, the runtime treats waiting like a tool return — creates the task, pauses and persists the run, resumes when the human acts.
4. **Is human review built in or bolted on?** — Built in. The runtime treats waiting on a human like waiting on a tool return — same primitives, different latency — so a human-in-the-loop step is first-class, enforced by the runtime regardless of prose.

### Compile (`/platform/compile`) — 4 Q
1. **How does the Compile flow work?** — An Operator states a goal; the Application Gateway invokes the compiler Harness, which surveys the workspace, optionally asks clarifying questions, plans the topology, emits the Harness as an OHM manifest, runs a review dialogue, and commits — registering triggers. Nothing runs until the Operator commits.
2. **Do I need to write code?** — No. Describe the goal in plain language and the Compile flow emits the Harness. Oraclous separates what work needs doing (prose, by the Operator) from how the runtime enforces it (code, written once by the platform).
3. **What is an Operator?** — The person who states an organisation's goal in natural language so the platform can compile it into a governed Harness. The Operator owns what needs doing; the platform owns how the runtime enforces it.
4. **Can I review the Harness before it runs?** — Yes. The Compile flow emits a manifest and runs a review dialogue before anything commits. You read and refine the exact governed object that will execute; only your commit makes it live and registers triggers.

### ReBAC governance (`/platform/rebac-governance`) — 5 Q
1. **What is ReBAC and how is it different from RBAC?** — ReBAC defines permissions by the relationships between people, Agents, workspaces, and data. RBAC defines them by static assigned roles and tends toward "role explosion." ReBAC scales with relationships instead of a growing role table; Oraclous enforces it platform-wide at the Harness level.
2. **How do you control what an AI agent can access?** — Each Agent has a capability allocation and runs inside a Harness under a versioned policy set. The Harness Runtime enforces the allocation, the policy set's budget caps, and ReBAC checks in code. Prose cannot expand the allocation — the Runtime refuses. Code wins.
3. **What are the five policy sets?** — Development default, Staging default, Production default, Production strict, Production federated. Each is named and versioned, declaring trust tier, signature requirement, BYOM constraints, capability allow/deny, audit level, retention, and budget ceilings (token, tool-call, wall-time). A Harness references one by ID.
4. **Is the enforcement deterministic, or does the model decide?** — Deterministic and coded. ReBAC checks, budget caps, HITL gates, and output redaction live in platform code, not the model's interpretation. Prose handles flexible reasoning; when prose contradicts a structured policy, the policy wins, every time.
5. **How do auditors get proof of access?** — Provenance is a universal sink on the Substrate: every result is traceable to where it came from and how it was produced, and every node, relationship, query, and audit log carries an `organization_id`. With versioned policy sets and ReBAC, auditors get an inspectable record, not a verbal assurance.

### BYOM (`/platform/byom`) — 4 Q
1. **What does "bring your own model" mean?** — You supply the LLM provider and credentials, and the platform runs against them rather than forcing one built-in model. The LLM is a resource an Agent uses, configured by you and resolved at agent, workspace, or organisation level — so model choice stays yours.
2. **Can I use Claude, GPT, Gemini, or a local model?** — Yes. BYOM supports three protocol shapes (Anthropic-native, OpenAI-compatible, Gemini-compatible) covering Claude, GPT-class models, Gemini, and any local/self-hosted model that speaks one of those wire protocols. Run several side by side in the same Harness.
3. **Where do my model keys live?** — In the Credential Broker, envelope-encrypted under your organisation's KMS-controlled wrapping key — never in the Harness, never in plaintext. Resolved per invocation, never cached outside the broker. In cloud mode the wrapping key is customer-controlled, so Oraclous cannot unilaterally decrypt them.
4. **Does switching models change the harness?** — No. The Harness describes what gets done; model config describes which engine reasons over it. Switching providers is a config change resolved at runtime — orchestration spec, actor roster, policy envelope, and task board stay unchanged. No agent code to rewrite.

### Knowledge graph (`/platform/knowledge-graph`) — 5 Q
1. **How does the platform store organisational memory?** — As a knowledge graph on the Substrate — nodes and relationships scoped by `organization_id`. Write side ingests text, documents, code, structured, and temporal data and runs analytics; read side answers semantic, full-text, hybrid, graph-traversal, and temporal queries. Every entry carries provenance.
2. **Is retrieval ReBAC-bounded?** — Yes. Every retrieval shape is bounded by ReBAC; an Actor retrieves only what its relationships permit. Retrieval is not a path around governance, and every result carries provenance.

Every node, relationship, query, cache entry, and audit log carries an `organization_id` every query must filter on. Federation is cross-workspace within one organisation, never cross-organisation.
5. **What data types can the graph ingest?** — Multi-modal ingestion of text, documents, structured data, code, and temporal data, with schema management and analytics (community detection, centrality). Each write goes through the provenance sink, so every node records its source.

### Human-in-the-loop (`/platform/human-in-the-loop`) — 5 Q
1. **How do humans approve agent work?** — At a HITL gate the Runtime resolves the assignee, creates a task on the workspace task board, notifies the person, and pauses the run with state persisted. The human approves, blocks, edits, or supplies input; the Runtime resumes from exactly where it paused.
2. **Is HITL a workaround or built in?** — Built in. A first-class runtime primitive — the Runtime treats waiting on a human like waiting on a tool return. Not a webhook or dashboard bolted on; a step in the same flow the Agents run in, and the run pauses there until a human acts.
3. **Can a human block an agent step?** — Yes. At a HITL gate the human can let the step proceed or block it. The gate is coded enforcement, so even if a manifest's prose tries to skip it, the Runtime enforces it — code wins. Work cannot continue past a block without human action.
4. **What happens to the run while it waits for a human?** — It pauses and its state is persisted. Nothing consumes budget while it waits, and the durable state survives a restart. When the human acts, the Runtime resumes from exactly where it paused — no lost context, no re-run from the top.
5. **Who decides where a human gate is required?** — The policy set the Harness runs under, plus the Harness's orchestration. Policy sets anchor where mandatory gates apply; the Runtime enforces them in code. Whether a step requires a human is governance, not a per-run judgement the model makes.

### Execution & scheduling (`/platform/execution-scheduling`) — 4 Q
1. **Can agents run on a schedule?** — Yes. A Harness declares triggers in its OHM manifest; the Execution Engine registers and fires them on time, so a recurring Agent wakes on schedule. Because the schedule lives in the governed Harness, the run is governed, metered, and audited like any other.
2. **What happens if a run crashes?** — Durable execution is checkpointed, so an interrupted run resumes from its last checkpoint rather than starting over. The Execution Engine keeps a durable execution context for exactly this — the same machinery that lets a run pause for a human or schedule and resume cleanly.
3. **Are there retry limits?** — Yes. Retries are bounded by the declared count in the Harness's policy set — and no more. With enforced timeouts, a failing step can't loop away unbounded or run up surprise cost. Retry behaviour is part of the governed policy envelope.
4. **Can I pause or cancel a running agent?** — Yes. Pause, resume, and cancel are first-class for durable executions. The machinery that lets a run pause for a human lets you intervene on a long-running job, while its checkpointed state stays intact.

A widget renders inside its own root, scopes its own styles, talks only to the configured gateway endpoint, and never reads or modifies the host DOM beyond a postMessage to a configured origin.

Oraclous exposes a standard MCP server through the Application Gateway, so Claude Desktop, Cursor, and Continue connect directly. The MCP tool adapter maps an MCP tool definition to an OHM tool; SKILL.md and OpenAPI 3.x adapters do the same. Imported tools become uniform capability descriptors — governed, versioned, invoked like native ones.

### Metering (`/platform/metering`) — 5 Q
1. **What does metering track?** — Five dimensions of consumption at the Substrate, per organisation and per workspace: tokens, tool invocations, storage (bytes), execution time, and cross-workspace traversals. Each is an authoritative usage event (`organization_id`, principal, action type, quantity, unit, dimensions, timestamp) on a durable, append-only stream.
2. **Does Oraclous set prices?** — No. The Substrate emits counts and bytes, never USD and never credits — both are rate tables belonging to a separable billing consumer. Metering measures neutrally and leaves pricing under your control. Under BYOM your model spend is whatever your provider charges.
3. **Can I enforce budgets?** — Yes — through the policy sets, not metering. Each versioned policy set declares budget ceilings (token, tool-call, wall-time caps per run), and the Harness Runtime enforces them in code: when a run exhausts its budget, the Runtime halts. Metering gives visibility; policy-set caps give a hard stop.
4. **How is metering different from billing?** — Metering owns the authoritative usage data; billing is a separable consumer, not its owner (ADR-009). The Substrate records consumption neutrally in counts and bytes; a downstream billing system applies a rate table. Keeping them separate keeps measurement honest and pricing yours.
5. **Is the usage data auditable?** — Yes. Usage events are written to a durable, append-only stream scoped by `organization_id` and attributed to a principal, alongside provenance on the same Substrate. One source of truth feeds operations dashboards, audit, and any billing consumer.

## Why-Oraclous pages

### Bespoke code is brittle (`/why-oraclous/bespoke-code-is-brittle`) — 4 Q
1. **Should I build my own agent pipeline or buy a platform?** — Build in code for one-offs or genuinely novel machinery. For operational work that changes often, buy a platform: a hand-built pipeline couples the goal to its enforcement, so every change is an engineering ticket. Oraclous lets you describe the goal in prose and change it without a rebuild.
2. **Why is bespoke automation so brittle?** — Each pipeline is hand-built for one use case, coupling what the work is to how it runs. Credentials, retries, audit, and access checks are rewritten per pipeline, so a small change touches code in several places. The brittleness is structural, not a code-quality problem.
3. **Do I still need engineers if I use Oraclous?** — Yes — for different work. Engineers stop hand-building a pipeline per use case and instead extend the platform's Capabilities and review compiled Harnesses. Operators describe goals in prose; engineers own the substrate, integrations, and novel machinery that genuinely needs code.
4. **Is "describe the goal in prose" just low-code with extra steps?** — No. Low-code still asks you to assemble the flow by hand; the brittleness moves to a canvas. Oraclous compiles a stated goal into a governed Harness through the Compile flow — survey, clarifying questions, topology planning, review dialogue — and you commit only after reviewing what it will do.

Oraclous provides these as substrate while keeping BYOM and open source, so you don't rebuild the control plane per use case.
4. **Do I lose low-level control by moving from a framework to Oraclous?** — You trade some line-by-line control over a single agent's internals for substrate you don't rebuild and governance you don't wire. If bespoke per-agent control is your real requirement, a framework may fit better. If your real cost is per-use-case plumbing, Oraclous removes it.

### Data sovereignty (`/why-oraclous/data-sovereignty`) — 5 Q
1. **What is data sovereignty for AI?** — The guarantee that your data stays under your control regardless of who operates the platform. For AI agents: knowing exactly what each Agent can reach, that cross-organisation flow is impossible, and that the vendor cannot decrypt your state. Oraclous guarantees this architecturally, not by policy.
2. **Can hosted vendors — including Oraclous — see my data?** — No. In cloud mode, customer state is encrypted under customer-controlled keys, and KMS separation means staff cannot decrypt it by virtue of operating the platform (ADR-008). Support and debugging happen with your participation.
3. **Who holds the encryption keys?** — You do. Customer state at rest is encrypted under key material the customer controls; the Credential Broker keeps credentials under per-organisation encryption that never leaves it in plaintext. In cloud mode, key-management separation means Oraclous cannot unilaterally decrypt your state.
4. **Is self-hosting required for data sovereignty with Oraclous?** — No. Self-hosted and cloud-hosted carry identical data-sovereignty guarantees — structural org isolation and customer-held keys — as an accepted architectural decision (ADR-008). The difference between modes is operational, not architectural.
5. **How is one organisation isolated from another?** — By an `organization_id` on every stored node, relationship, query, cache entry, and audit log, enforced at the Substrate. Every query must filter on it, so cross-organisation data flow is structurally impossible — no admin override can cross it. Federation allows controlled cross-workspace requests under ReBAC, never crossing the organisation boundary.

### Vendor lock-in (`/why-oraclous/vendor-lock-in`) — 5 Q
1. **How do I avoid vendor lock-in with AI agents?** — Avoid lock-in on three axes: the model, the platform, and the exit. Choose model freedom (BYOM), open-source/self-hostable code, and a portable, vendor-neutral unit of work you can export and re-run. Oraclous gives all three through BYOM, platform-as-code, and OHM portability.
2. **Can I switch LLM providers without a rewrite?** — Yes. With BYOM the model is a resource the Agent uses, not the Agent itself, and config resolves at agent, workspace, or organisation level across Anthropic-native, OpenAI-compatible, and Gemini-compatible shapes (ADR-007). Change the model by config; the Harness stays the same.
3. **Is Oraclous really open source, or open-core?** — Open source and platform-as-code — the substrate's machinery is code, deployed and versioned through normal engineering practice, so it is inspectable, forkable, self-hostable. Read the architecture and ADRs before committing, and run the whole thing yourself with no vendor in the loop.
4. **What does portability not cover?** — Four things don't travel in an OHM export: knowledge-graph data (via Neo4j/RDF/JSON-LD), the member directory and ReBAC graph (platform-internal), credentials (never leave the broker in plaintext), per-Actor Consciousness records (substrate-anchored). Documented plainly so there are no exit surprises.
5. **How is BYOM different from a platform "supporting" many models?** — Many platforms support several models but still bind your agent code to one vendor's interface. BYOM treats the LLM as a swappable resource resolved by config, so you move between providers — or run them side by side — without rewriting the Harness, and provider keys stay encrypted in the Credential Broker.

### Agent governance & audit (`/why-oraclous/agent-governance-audit`) — 5 Q
1. **How do you govern AI agents in an enterprise?** — Relationship-based access (ReBAC) enforced at the platform level, guardrails in code that prose can't override, versioned policy sets with hard budget caps, and a provenance trail. Oraclous does all four, governing human Operators and AI Agents under one access model rather than two systems.
2. **What is ReBAC vs RBAC?** — RBAC grants access by an assigned role and suffers "role explosion" as exceptions accumulate. ReBAC grants access by an entity's relationship to the specific resource. Oraclous enforces ReBAC platform-wide at the Harness level, which fits the "this Agent, for that Operator, on this data" question agents raise.
3. **Can AI agents respect a company's existing permissions?** — Yes. Because access is ReBAC at the Harness level and humans and Agents are symmetric Actors under one access model, an Agent inherits permissions from its relationships — it can only reach what its relationship to the resource allows, and coded policy overrides any prose that tries to widen that.
4. **Where are the audit logs for AI agents?** — In provenance — a universal sink on the Substrate recording where every result came from and how — alongside the ReBAC graph and the policy set each run executed under. Audit level (summary to full) and retention (30 to 730 days) are declared by the policy set, so you control depth and duration.
5. **Can an AI agent talk its way past its limits?** — No. Two-layer enforcement: flexibility is in the prose an Agent reads, but limits (capability allocations, budget caps, HITL gates) are in code enforced by the Runtime. A prose instruction that tries to bypass a gate or expand permissions is refused regardless of wording. Code wins.
